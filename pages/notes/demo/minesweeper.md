## 日志

### 胜负判定条件

扫雷的一个胜利条件是翻开棋盘上所有非雷的格子，如果翻到有雷的格子则游戏失败。

游戏过程中若玩家断定某个未翻开的格子是雷时，可以通过插旗进行标记，如果刚好把都所有未翻开的雷全部插上旗，也算游戏胜利。

### 鼠标操作交互

游戏支持以下三种鼠标操作：

| 鼠标 | 未翻开的格子               | 已翻开的格子                                                             |
| ---- | -------------------------- | ------------------------------------------------------------------------ |
| 单击 | 有插旗就取消插旗，否则翻开 | 高亮周围（至多 8 个格子）未翻开的格子                                    |
| 右击 | 在 插旗/取消插旗 之间切换  | 同上                                                                     |
| 双击 | 高亮格子自身               | 如果有数字且该数字刚好等于周围插旗的数量，那么会翻开周围所有未插旗的格子 |

> ps：话说双击这个操作我刚开始还以为是同时点击鼠标左右键... 😓

通过监听 `click`、`contextmenu`、`dblclick` 事件，可以很轻松的处理这些鼠标交互，同时为了适配移动端的触屏滑动，我额外加了两个选项。

另外当按下鼠标在格子上滑动时，还需要有高亮格子的提示效果，这个功能可以通过监听 [Pointer Event 指针事件](https://developer.mozilla.org/zh-CN/docs/Web/API/Pointer_events) 来实现，这样在触屏上滑动时也能生效。

### 开局布雷算法

> [!WARNING] ❓
> 如何将 N 个雷均匀放置在 M 个格子上，其中 M >= N

咋一看这是个很简单的问题，我最开始的方案是：

1. 为 M 个格子设置编号
2. 从所有号码中随机抽取一个，记录后不要“放回”
3. 在剩下的号码中继续执行步骤 2，这样反复执行 N 次后，抽到的所有号码就是布雷的区域

这种做法可以保证雷是随机分布在地图上各个小块中。但它的游戏体验好吗？

嗯，玩家可能会在翻开第一块后就 G 了，这样的话游戏就太不友好了。很容易想到，我们可以在上述方案的步骤 2 上添加一个前置判断：

- 如果这是第一次抽取（表现为玩家开局后翻开第一块格子），得到的编号需要“丢弃”（既不记录，也不放回）

用来规避开局就炸的极端情况。但多玩几次后，我发现这种做法还是有点问题的，特别是当游戏难度变高后，很容易出现开局后连续翻开几块都没有触发“自动翻开逻辑”，这样玩家翻下一块时还是需要靠运气猜。所以这里再重新修改一下：

- 如果这是第一次抽取（表现为玩家开局后翻开第一块格子），将这个编号和周围相邻的编号（至多 8 个）全部“丢弃”（既不记录，也不放回）

这样就可以保证至少触发一次“自动翻开逻辑”，这也是我目前的做法。不过个人感觉更好的做法在确保给到玩家一个良好的“局面”后（比如这个局面可以简单理解成玩家至少可以确认一个一定不是雷的格子等等），再让玩家有几率翻到雷。可惜能力有限，我并没有在游戏中实现这一步。嗯，如何设计和评估这个“局面”，也是我认为扫雷游戏设计的难点和魅力所在。

### 自动翻开逻辑

这个可以看作一种对玩家的奖赏机制：当玩家翻开一块雷，且周围格子都不是雷时，就会触发该奖赏：自动翻开周围所有小格，从而帮玩家节省操作时间。

另外在自动翻开周围所有小格时，如果其中有小格也满足周围区域都不是雷，还要继续触发。故我们可以用递归方式去处理，具体实践中需要注意递归终止条件，避免出现死循环。

## 后记

其实从上面的操作说明中可以看到，扫雷的鼠标交互还是挺丰富的。微软当年（1992 年）将扫雷内置在 windows 3.1 系统中发布，也希望用户通过扫雷来熟悉鼠标操作。

如今公众日常使用最广泛的人机交互应该算是触屏方式，但移动端的触屏并不能很好模拟完整的鼠标操作（比如右击和双击）。相比之前的 2048，扫雷并不适合在触屏上玩，这也是我在复刻游戏时的感想。

游戏寓教于乐的方式可以帮助用户快速熟悉新的交互逻辑，比如本文中提到的扫雷（对应鼠标操作），2048（对应触屏滑动），以及当年 iPhone4 发布会上乔布斯曾用玩游戏的方式来演示过陀螺仪。那么今年（2023 年）苹果公司新发布的 Vision Pro，是否又能给大众带来一套全新的交互体验呢？
